## Page-byte encoding convention — RESOLVED via real SAVE evidence

I independently verified by reading CHOMPER from `/Users/pmoore/Downloads/GoodSamC2/x.mgt`
(a real SAM-saved BASIC file at PROG=0x5CD5):

- **CHOMPER body header bytes 3-4 (PageOffset) = `D5 9C`** (= 0x9CD5, **BIT 15 SET**)
- **CHOMPER body header byte 8 (StartPage) = `0x00`** (= `(0x5CD5 >> 14) - 1`)
- **CHOMPER dir 0xED-0xEE (PageOffset mirror) = `D5 9C`** (bit 15 set)
- **CHOMPER dir 0xEC (StartPage mirror) = `0x00`**

So **canonical real-SAVE convention is**:
- Page byte = `(addr/16384) - 1` (NOT `addr/16384` as audit and sam-paging.md claimed)
- Offset = `(addr & 0x3FFF) | 0x8000` (bit 15 SET)

This refutes:
- Audit-agent's claim that "ROM `HDRNMS` writes page byte = `addr/16384`" — `HDRNMS` is for the BASIC prog-length triplets (HDR+HDN+0..+8), NOT for the start-address mirror.
- `sam-paging.md:765`'s claim that bytes for 0x6000 should be `01 00 A0`.

samfile's encoder (`samfile.go:485-486`) matches the page byte but skips bit-15 (writes `<page>, <offset_LSB>, <offset_MSB>` with no `| 0x80`). samfile is the source of the page-byte convention since Pete built it from Tech Manual.

**Right encoding for stub at `LOAD_ADDR=0x6000`**: bytes `00 00 A0` (page=0, offset=0xA000 with bit 15 set).

This decodes correctly via samfile's decoder (showing `Start: 24576`) AND matches what real ROM SAVE would write. Best of both worlds.

**Action when applying fixes**:
- Body header bytes 3-4 (PageOffset): keep bit 15 set form. For stub/IN that's `0x00, 0xA0`.
- Body header byte 8 (StartPage): leave as `(LOAD_ADDR >> 14) - 1` (already correct in original code).
- Dir entry 0xEC-0xEE: write `[(LOAD_ADDR>>14)-1, LOAD_ADDR & 0xff, ((LOAD_ADDR>>8) & 0x3F) | 0x80]` — for stub/IN that's `[0x00, 0x00, 0xA0]`.
- Body header bytes 5-6: write `FF FF` (cosmetic, matches CHOMPER + Tech Man L4293).

Audit recommendation #10 had the page byte WRONG (used `addr>>14` = 1 instead of `(addr>>14)-1` = 0). I caught this empirically when samfile reported `Start: 40960` instead of `24576`. The corrected encoding above gives `Start: 24576` correctly.
- **Why does the full stub fail?** The most likely candidates per
  `sam-file-io.md`:
  - UIFA bytes 31-36 not populated for HOFLE (output file metadata).
  - HERAZ should be called before HOFLE to clear pre-existing OUT.
  - RST 8 / hook number / register convention deviation.
- **Why does CLEAR crash AFTER the AUTO file fix?** The fix changed
  the visual crash pattern but didn't eliminate the crash. Some other
