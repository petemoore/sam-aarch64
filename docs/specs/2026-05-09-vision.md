# Vision: aarch64 development environment on the SAM Coupé

**Status**: approved 2026-05-09. Cross-cutting; does not change phase-to-phase.

## What we are building

A single Z80 program that runs on a SAM Coupé and combines three subsystems:

1. A **visual editor** for aarch64 source code, with structured input —
   the editor knows aarch64 syntax and produces a tokenised binary source
   format directly (no plain-text parsing on the SAM).
2. An **aarch64 assembler** that consumes that binary source and emits a
   flat aarch64 binary suitable for direct loading on a Raspberry Pi
   (`kernel8.img`-style — no ELF wrapping needed).
3. A **TFTP server** that serves the assembled binary, plus Pi firmware
   blobs read from the SD card on the Quazar Trinity, directly over
   ethernet to a Raspberry Pi on the same LAN.

The Pi netboots a kernel built on the SAM. The development loop closes
without leaving the SAM Coupé.

## Why

Pete writes bare-metal aarch64 in his `~/git/spectrum4` project
(a Raspberry Pi 400 kernel). The motivation here is to make a SAM Coupé
the dev box for that work, end-to-end — for the love of it, and as a
self-hosting retro-computing project.

## Hardware

- SAM Coupé, 512K RAM (256K stock + 256K upgrade).
- Quazar Trinity interface: ENC28J60 ethernet, 128K EEPROM, MMC/SD card
  slot. Ports `&DC`–`&DF`. SPI shape: every byte written returns one byte
  read (with one-byte read-lag).
- Raspberry Pi 400 on the same LAN as the SAM. Pi netboots over TFTP.

## Native source format

Source files on the SAM are **binary-tokenised**, COMET-style — not plain
ASCII. Each `.s` file is a stream of length-prefixed records (instruction,
label, directive, comment, structural marker). The editor produces this
format directly; the assembler consumes it; humans never see it.

Equivalent in expressiveness to GNU `as` aarch64 source. A lossless text↔
binary mapping is provided by Mac-side `text2bin` and `bin2text` tools.
The repository's source-of-truth for any aarch64 source code (test
fixtures, examples) is plain text — the binary form exists only on the
SAM during editing.

## Validation strategy

Every aarch64 instruction we emit is round-tripped against
`aarch64-none-elf-as`: the same plain-text source assembled by both tools
must produce byte-identical `.text` output. Any divergence is a bug in
our assembler. CI runs the round-trip suite on every commit.

## Phases

Each phase delivers an independently useful slice. Each phase has its own
spec → plan → implementation cycle.

- **Phase 1 — Standalone assembler.** Source from disk → flat binary,
  listing, symbol map to disk. No editor. No network. See
  `2026-05-09-phase1-assembler.md`. *Riskiest piece de-risked first; the
  encoder is the algorithmically novel work.*

- **Phase 2 — Visual editor.** A SAM-native structured editor that
  produces the tokenised binary source format. Replaces "write `.s` on
  Mac, `text2bin`, transfer to SAM" with on-SAM editing.

- **Phase 3 — TFTP server, possibly with SD-card-served firmware.**
  Trinity ethernet driver lifted from `~/git/trinload`, IP/UDP layered
  on top, TFTP framing on top of that. Optionally serves Pi firmware
  files (`bootcode.bin`, `start*.elf`, `fixup*.dat`) directly from SD
  card on the Trinity, in addition to the in-memory assembled kernel.

- **Future — SAM as daily-driver workstation.** A VT100 terminal app
  running on the SAM, connecting over TCP to a Mac-side bridge daemon
  that handles SSH/auth/encryption. Lets Pete use the SAM as a regular
  computer day-to-day. Implies the IP stack should be designed with
  TCP support in mind from Phase 3, even if Phase 3 itself only needs
  UDP. Strong candidate for Phase 3 networking: Adrian Brown's Z80
  port of Adam Dunkels' uIP, which already implements TCP/IP on the
  Trinity.

## Development environment

- **Source-of-truth**: this repo, under git on the Mac. Plain-text Z80
  assembly for our tool's own code.
- **Build tool**: pyz80 (Simon Owen's modern Z80 assembler).
- **Test runtime**: SimCoupé in headless batch mode for CI; real SAM for
  periodic spot-checks.
- **CI**: GitHub Actions running the full layered test suite on every
  commit.
- **File transfer**: `samfile` (Pete's Go tool) for adding/extracting
  files to/from `.mgt` SAM disk images.

## Reused upstream tooling

- `~/git/trinload` (Simon Owen) — ENC28J60 driver + IP/UDP stack, lifted
  for Phase 3.
- `~/git/comet2txt` (Simon Owen) — already used to detokenise the COMET
  source; informs the design of our own binary format.
- `~/git/samfile` (Pete) — disk-image manipulation for the test harness.
- pyz80, SimCoupé (Simon Owen).
- ARM Machine Readable Architecture (ARM MRA) XML — official source of
  every aarch64 encoding form. Filtered to the project's instruction
  subset and compiled into encoder tables.

## Out of scope (any phase)

- ELF object file output. The Pi loads raw bytes; ELF would be wasted
  effort.
- Linking. Each `.s` file is a complete, self-contained unit (Z80
  tradition; no linker on the SAM).
- Multi-file projects with separate compilation. `.include`-style
  inclusion may be considered for Phase 2+ but is not architectural.
- Full TCP stack and SSH crypto (deferred indefinitely; the future
  daily-driver story uses a Mac-side SSH bridge instead).
